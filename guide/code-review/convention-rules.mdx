Convention Rules keep CloudThinker's Code Review aligned with how your team actually writes code. It works in two ways:

- **The reviewer follows your existing conventions.** It loads the convention files already in your repository on demand, so it reviews against the same standards your coding assistants already use.
- **The reviewer learns from past reviews.** The issues your team keeps running into, and keeps fixing, become reusable rules that the reviewer applies to future PRs on its own.

---

## How it works

- **Your existing convention files.** Point the reviewer at the coding-assistant convention files your team already maintains, such as `CLAUDE.md`, `.claude/rules/`, `AGENTS.md`, or `.cursorrules`. The reviewer loads them on demand during each review, so there's nothing to author.
- **Rules learned from past reviews.** On a schedule you choose, CloudThinker looks at the review findings your team has already resolved on a repository, turns the recurring ones into named rules, and applies those rules in later reviews. You manage the results from the **Rules** tab.

The two work independently or together.

### Custom convention paths

You can add **Custom Convention Paths** to point the reviewer at additional convention files or directories in your repository. This is useful when your team keeps convention files in non-standard locations or uses more than one tool.

Click **+ Add path** in the Convention Rules dialog to specify relative paths to additional convention files or directories.

## Rules learned from past reviews

CloudThinker learns your team's coding conventions from each repository's default branch and applies them in future reviews. As your team resolves review findings, the recurring ones become named rules. New rules show up in the **Rules** tab on their own after the next scheduled run, so the reviewer keeps catching the issues your team cares about without anyone writing rules by hand.

<Info>
  Until the first run finishes, the **Rules** tab is empty. Rules appear on their own after the next scheduled run for the repository, once it has enough resolved review findings to learn from.
</Info>

### Set the update schedule

Rule updates run per repository on a cadence you choose. In **Code Review → Repositories**, use the **Convention Rules** column to pick how often CloudThinker refreshes a repository's rules:

{/* IMAGE TODO: screenshot of the Repositories table Convention Rules column showing the Off / Weekly / Monthly dropdown. Path to be provided. */}

| Option | Behavior |
|--------|----------|
| **Off** | Don't update rules for this repository |
| **Weekly** | Refresh rules once a week |
| **Monthly** | Refresh rules once a month |

The schedule becomes available once Code Review is enabled for the repository and the repository has finished cloning. Convention Rules work with **GitHub, GitLab, Bitbucket, and Azure DevOps** repositories. **AWS CodeCommit is not supported**, so the schedule stays disabled for CodeCommit repositories.

### The Rules tab

Rules appear in the **Rules** tab under **Code Review**. You don't write rules here; the tab is where you review them, filter the list, and turn individual rules on or off.

{/* IMAGE TODO: screenshot of the Rules tab table (Rule, Repository, Cited, Enable columns with the search / repository / status filters). Path to be provided. */}

| Column | What it shows |
|--------|---------------|
| **Rule** | The rule's name and a short identifier (slug) |
| **Repository** | The repository the rule came from |
| **Cited** | How many past review findings referenced this rule |
| **Enable** | A switch to turn an individual rule on or off |

Over time a repository builds up rules like these, drawn from the issues that came up most in its own reviews:

| Rule | Cited |
|------|-------|
| Money values stored or computed as floating-point dollars instead of integer cents (`money-float-arithmetic`) | 52 |
| Commented-out code committed to source (`commented-out-dead-code`) | 38 |
| Server actions exported with no authentication or authorization check (`server-action-missing-auth`) | 25 |

Keep an eye on the **Cited** count. It shows which conventions your team runs into most, so you can tell at a glance what matters and decide which rules to keep, fix at the source, or turn off.

Filters at the top of the table help you find rules:

- **Search** by slug or title.
- **Repository** to focus on one or more repositories.
- **Status** to show only **Active** or **Disabled** rules.

Turn off any rule you don't want the reviewer to apply, and turn it back on later at any time.

## Permissions

Managing convention rules requires the appropriate workspace permission. Without it, the Rules tab reports that you don't have permission to view convention rules and to contact your workspace admin. Ask a workspace admin to grant access if you need to view rules or change a repository's schedule.

## FAQ

<AccordionGroup>
  <Accordion title="Do I need convention files if CloudThinker learns rules for me?">
    No. The two work independently, so you can use either or both. Convention files are read right away; learned rules build up from your reviews over time.
  </Accordion>
  <Accordion title="Why don't any rules appear yet?">
    Rules show up after the first scheduled run for a repository. Make sure Code Review is enabled, the repository has finished cloning, and the Convention Rules schedule is set to Weekly or Monthly rather than Off. A new repository also needs enough resolved review findings before the first rules appear.
  </Accordion>
  <Accordion title="Why is the schedule dropdown greyed out?">
    The schedule turns on once Code Review is enabled and the repository has finished cloning. It stays disabled for AWS CodeCommit, which isn't supported.
  </Accordion>
  <Accordion title="Can I stop a rule without losing it?">
    Yes. Turn the rule off in the **Enable** column. It stops applying to reviews right away, and you can turn it back on at any time.
  </Accordion>
</AccordionGroup>
