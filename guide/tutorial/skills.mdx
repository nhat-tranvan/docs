## Why custom skills

Default agents arrive knowing the *public* shape of code review, incident response, and cloud operations — what good looks like in general. They do not know what *good* looks like in **your** codebase, **your** runbooks, **your** team's review standards. Without that knowledge, agents fall back to generic best practices and the team falls back to typing the same context into every conversation: *"remember our naming convention,"* *"check the deprecation list,"* *"follow the incident escalation rule."*

A custom skill externalizes that context once and applies it on every relevant loop. Senior expertise becomes a team-wide artifact instead of tacit knowledge held by one engineer. New hires get the same review quality as the founder; on-call responders get the same incident discipline as your most experienced SRE.

Common categories teams encode as skills:

| Domain | Examples |
|---|---|
| **Code review** | Naming conventions, API design rules, language-specific lint patterns, security review checklist, performance hot-spot patterns |
| **Incident response** | Escalation policies, runbook selection rules, service-specific debugging order, communication templates |
| **Cloud operations** | Account-tag conventions, environment promotion rules, cost-attribution policies, region preferences, compliance constraints |
| **Documentation** | Style guide, terminology, voice rules, audience-specific framing |

A useful heuristic: if you find yourself typing the same instruction into more than two prompts, that instruction belongs in a skill.

## What You'll Set Up

By the end of this tutorial, you'll have one custom skill created, enabled, and assigned to a feature — applied automatically every time the agents run a task in that feature, with no extra prompting from you.

<Steps>
  <Step title="Navigate to Skills">
    Open **Skills** from the sidebar — it sits in the main workspace navigation alongside *New chat* and *Artifacts*. This is where every skill in the workspace is created, edited, and toggled on or off.
  </Step>
  <Step title="Create a Skill">
    Click **New Skill** and pick a creation method:

    - **Create with AI** — describe what you need in plain language; the agent generates a structured SKILL.md
    - **Write skill instructions** — author the skill manually using the form

    For your first skill, choose **Create with AI**. Describe one concrete rule your team applies — a naming convention, a review checklist item, a deprecation warning — and let the agent draft it.
  </Step>
  <Step title="Review the Generated Skill">
    The new skill appears as a card on the Custom Skills page. Click into it to open the detail view: file tree on the left, rendered markdown on the right.

    Read the instructions end to end. Skills that read clearly to a human read reliably to an agent — if a rule is ambiguous to you, it will be ambiguous to the model.
  </Step>
  <Step title="Enable the Skill">
    Toggle the skill **on** with the switch on the card. Disabled skills are saved but never applied — useful for drafting and staging changes without affecting live agent behavior.
  </Step>
  <Step title="Assign to Features">
    Use the feature checkboxes on the skill card to control where the skill applies:

    - **Code Review** — applied during PR analysis
    - **Incident** — applied during incident investigation
    - **Assessment** — applied during infrastructure assessments

    Assign the skill only to the features where it actually belongs. A code-review naming convention does not need to load during incident triage.
  </Step>
  <Step title="Verify It Works">
    Trigger a task in one of the assigned features — for example, open a PR if the skill is assigned to Code Review. Inspect the agent's output and confirm the rule shows up in its reasoning and comments. If it doesn't, the instruction was probably too abstract; rewrite with concrete examples and retest.
  </Step>
</Steps>

## How skills compound with the loop

Every CloudThinker module runs the [agentic loop](/guide/tutorial/vibeops) — **Detect → Analyze → Resolve → Validate**. Skills join the loop at the *Analyze* phase: when the agent forms a hypothesis or plan, it consults the skills assigned to the active feature alongside its base behavior. The effect across many iterations:

- Your review rules apply consistently to every PR, including the ones reviewed at 4 PM on a Friday
- Your runbooks guide investigation order on every incident, regardless of which engineer triggers the loop
- Your conventions show up in agent output without anyone restating them

Skills are how the platform adapts to your environment without retraining. The encoding happens once; the application happens every time.

## Writing effective skills

A skill is a Markdown file with YAML frontmatter:

```markdown
---
name: api-review-standards
description: Enforce API design standards during code review
---

## Naming

- Use plural nouns for collection endpoints (`/users`, not `/user`)
- Use kebab-case for multi-word paths (`/user-settings`)

## Error Handling

- Return a consistent error shape with `code`, `message`, `details`
- Use 4xx for client errors and 5xx for server errors only
```

Practices that make skills reliable in production:

- **Be specific.** Concrete rules with examples beat vague guidelines. *"Use kebab-case for multi-word paths"* outperforms *"Follow our naming style."*
- **Scope tightly.** One skill per domain. Separate security review from performance review from style — narrow skills are easier to maintain and debug.
- **Lead with examples.** Agents follow demonstrated patterns more reliably than abstract explanations.
- **Write only what the model doesn't already know.** Every token in the skill costs latency and budget; don't re-encode public best practices.

## Tips

- **Start with one skill.** Encode your most-repeated piece of context first. Watch how the agent applies it before adding more.
- **Iterate with AI.** Use **Edit with AI** on any skill to refine it through conversation — adding examples, removing ambiguity, sharpening rules.
- **Test on one feature first.** Enable a new skill on one feature, review the output across a few real runs, then assign it more broadly.
- **Treat skills as code.** Review them like you'd review a config change — clear diffs, named owners, rollback when something regresses.
