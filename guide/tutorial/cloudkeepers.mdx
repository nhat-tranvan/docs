## What You'll Set Up

CostOps is an autonomous pilot that continuously scans your cloud environment for cost waste — without waiting for a quarterly review. By the end of this tutorial, you'll have cost monitoring running on a schedule, findings ranked by impact, and alerts routed to the channels your team actually reads.

<Steps>
  <Step title="Navigate to CostOps">
    Go to [**Infrastructure** > **CloudKeepers**](/guide/infrastructure/cloudkeepers) in your workspace.

    Select the **CostOps** pilot. It monitors spending anomalies, idle resources, and optimization opportunities across your connected cloud accounts.

    <Note>
      CostOps is one of several autonomous pilots that share the CloudKeepers surface. SecurityOps (continuous misconfiguration and vulnerability scanning) runs in the same place but is covered separately.
    </Note>
  </Step>
  <Step title="Configure the Pilot">
    Set how the CostOps pilot runs:

    1. **Schedule** — how often the pilot scans (e.g., daily, every 12 hours). Daily is a sensible starting cadence.
    2. **Scope** — which cloud accounts and regions to monitor. Start with one account if you want to tune thresholds first.
    3. **Thresholds** — what qualifies as alert-worthy. A common starting threshold is *resources with cost > $50/month idle for 7+ days*.

    Toggle the pilot **On** to activate it.
  </Step>
  <Step title="Configure Alert Channels">
    Decide where findings get delivered:

    - **In-app** — findings appear in the CostOps dashboard (always on)
    - **Email** — sent to assigned team members
    - **Slack** — routed to a specific channel (e.g., `#cloud-cost`, `#finops`)

    You can route different severity levels to different destinations — Critical findings to Slack for immediate visibility, Low findings to in-app only to avoid noise.
  </Step>
  <Step title="Review Findings">
    After the first scan completes, open the **CostOps** dashboard.

    Each finding shows:
    - **Description** — what was detected
    - **Severity** — Critical, High, Medium, Low
    - **Resource** — the affected resource with a direct link
    - **Recommendation** — what to do about it
    - **Projected savings** — monthly impact if remediated

    For each finding you can:
    - **Save** — keep it for tracking and remediation
    - **Ignore** — dismiss it (a known exception or accepted spend)
    - **Delete** — remove the finding entirely
  </Step>
  <Step title="Take Action on Findings">
    For findings you want to act on:

    1. Click the finding to open the full detail view
    2. Review the **remediation playbook** with impact analysis
    3. Use **Custom Prompt** to ask the agent for more context — for example, *"show usage trends for this instance before recommending a rightsize"*
    4. Click **Implement** to apply the fix with agent assistance under whatever [approval policy](/guide/approval) you've set

    <Note>
      CostOps scans across all permitted cloud resources — including ones you haven't manually discovered. It catches waste in resources you may not know exist.
    </Note>
  </Step>
</Steps>

---

## Tips

- **Start with a daily scan.** Increase frequency once thresholds are tuned and false-positive rates are low.
- **Route Critical findings to Slack.** Real-time visibility on high-impact waste shortens the time to remediation.
- **Tune weekly.** Ignore false positives to improve signal-to-noise over time. The pilot learns which patterns you treat as accepted spend.
- **Pair with [SecurityOps](/guide/infrastructure/cloudkeepers).** Same pilot pattern, applied to compliance drift and vulnerabilities.
