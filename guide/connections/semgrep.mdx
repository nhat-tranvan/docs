Connect your Semgrep account to enable CloudThinker agents to scan code for security issues, triage findings from the Semgrep AppSec Platform, and inspect rules and code structure.

Semgrep authenticates with a single **access token** from the Semgrep AppSec Platform.

<Note>
Semgrep's paid **Team** and **Enterprise** tiers aren't supported yet. Support is planned for a future update.
</Note>

---

## Supported Platforms

| Platform | Support |
|----------|---------|
| **Semgrep AppSec Platform** | Cloud (`semgrep.dev`) |

---

## Prerequisites

- A **Semgrep AppSec Platform account**.
- An **access token** with API/web access so the agent can read cloud findings.

<Info>
Cloud findings require a token with **API / web access**. Org-wide policy and scan configs may additionally need **Agent** or **CI** scope, depending on how your Semgrep organization is set up.
</Info>

---

## Setup

<Steps>
  <Step title="Open Semgrep">
    Sign in to the [Semgrep AppSec Platform](https://semgrep.dev).
  </Step>
  <Step title="Create an Access Token">
    On the sidebar, click **Settings**, then on the top bar click **Tokens** (next to **General**). Under **API Tokens**, click **Create new token**:
    - **Name**: `cloudthinker`
    - **Scopes**: enable both **Agent (CI)** and **Web API**

    Copy the **Secret** value — this is your `SEMGREP_APP_TOKEN` — then click **Save**.
  </Step>
  <Step title="Add Connection in CloudThinker">
    Navigate to **Connections → Semgrep** and enter:
    - **App Token**: the access token you just created

    Click **Connect**. CloudThinker verifies the token and shows a **Connected** status.
  </Step>
</Steps>

<Warning>
Copy the access token immediately after creation. You'll need to create a new token if it's lost.
</Warning>

## Required Permissions

Grant the token the access CloudThinker needs to read your Semgrep data:

- **API / web access** — required to read cloud findings from the AppSec Platform
- **Agent / CI scope** — may be needed to read org-wide policy and scan configurations

<Tip>
Follow least privilege: start with a token that can read findings, and add scope only if a workflow needs org policy or scan configs.
</Tip>

---

## Agent Capabilities

Once connected, agents can:

| Capability | Description |
|------------|-------------|
| **Discovery** | Summarize the connected account and available scan capabilities |
| **Code Scanning** | Scan provided code for security issues using Semgrep rules |
| **Cloud Findings** | Pull and triage findings from the Semgrep AppSec Platform |
| **Rule Inspection** | Inspect rule schema to understand and refine detection rules |
| **AST Inspection** | Inspect the abstract syntax tree of code for deeper analysis |

### Verify the Connection

```bash
@oliver pull my open Semgrep findings and group them by severity
```

### Example Prompts

```bash
@oliver list open Semgrep cloud findings and #flag anything critical
@oliver scan this file for security issues with Semgrep
@oliver group my Semgrep findings by severity and summarize the top risks
```

## Troubleshooting

<Accordion title="Cloud findings fail to load">
The token is missing, invalid, or lacks API access. Create a new Semgrep access token with API / web access and reconnect.
</Accordion>

<Accordion title="Scan returns no findings">
The config or language may not match, or an OSS scan can't compute Pro results. Try a specific config such as `--config p/python`, or pull cloud findings for CI-computed Pro results.
</Accordion>

<Accordion title="Local scan rejected">
Scanning a local path is opt-in. Prefer an inline scan of provided code, or set `SEMGREP_ALLOW_LOCAL_SCAN=1` to allow local path scans.
</Accordion>

---

## Security Best Practices

- **Least-privilege scope** - Grant only the access CloudThinker needs, starting with read access to findings
- **Dedicated token** - Use a token named for CloudThinker so it's easy to audit and revoke
- **Token rotation** - Rotate the access token regularly
- **Revoke when unused** - Remove the token in Semgrep if you stop using the connection
