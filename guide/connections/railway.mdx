Connect your Railway account to let CloudThinker agents inspect projects, review deployment health, triage logs, and recommend next steps for Railway services.

Railway uses **OAuth**, so you do not need to create a Railway token, API key, project token, or local Railway CLI login for this CloudThinker connection.

---

## Prerequisites

- A **Railway account** with access to the workspace or projects you want CloudThinker to inspect.
- Permission to authorize CloudThinker through Railway's OAuth flow.

<Info>
OAuth scopes the connection to the Railway workspace or projects you authorize. Connect with the least-privileged Railway account that works.
</Info>

---

## Setup

<Steps>
  <Step title="Open CloudThinker">
    Navigate to **Connections → Railway** in your CloudThinker workspace.
  </Step>
  <Step title="Start the OAuth flow">
    Click **Connect** to open Railway's authorization page.
  </Step>
  <Step title="Authorize CloudThinker">
    Sign in, choose the Railway workspace or projects CloudThinker should access, and approve access.
  </Step>
  <Step title="Return to CloudThinker">
    Return to CloudThinker and wait for the connection test to finish.
  </Step>
</Steps>

<Note>
There are no environment fields to fill out for the Railway connection.
</Note>

---

## Required Permissions

CloudThinker requests these Railway OAuth scopes:

| Scope | Why CloudThinker needs it |
|-------|---------------------------|
| `openid` | Identify the signed-in Railway user |
| `profile` | Read basic profile information |
| `email` | Associate the connection with the authorizing user |
| `offline_access` | Keep the connection active after the browser flow |
| `workspace:member` | Access authorized Railway workspace resources |

Write operations, such as redeploys or Railway Agent workflows, require matching Railway access **and** explicit [approval](/guide/approval) in CloudThinker.

---

## Agent Capabilities

Once connected, agents can:

- List Railway workspaces, projects, environments, and services.
- Review deployment health and identify failed or stuck deploys.
- Inspect recent service or deployment logs.
- Summarize project inventory.
- Audit environment variables for missing production values.
- Recommend next steps for failed deployments.
- Trigger mutating Railway actions only after explicit user approval, such as redeploys or Railway Agent workflows.

### Example Prompt

```bash
@tony please list my Railway projects and services, then summarize any recent failed deployments without making changes.
```

## Troubleshooting

<Accordion title="No projects visible">
The OAuth flow may not have authorized the needed workspace or project. Reconnect Railway and select the correct workspace or project.
</Accordion>

<Accordion title="CloudThinker asks for RAILWAY_TOKEN">
You may be following the wrong setup path. Use CloudThinker's Railway OAuth flow instead of Railway tokens, API keys, project tokens, or local CLI login.
</Accordion>

<Accordion title="Logs cannot be fetched">
The agent may need a deployment ID, or both service and environment IDs. First ask Tony to discover projects, services, and deployments.
</Accordion>

<Accordion title="Connection succeeds but Tony cannot act on a project">
The Railway OAuth scope or workspace access may not match the project. Reconnect with the right Railway account and workspace.
</Accordion>

<Accordion title="Mutating action blocked">
CloudThinker requires explicit approval for writes. Review the exact action and approve only if it is correct.
</Accordion>

---

## Security Best Practices

- **Least-privilege account** - Authorize only the Railway workspace or project access CloudThinker needs
- **Approval for writes** - Keep redeploys, Railway Agent workflows, and other mutating actions approval-gated
- **Start read-only** - Verify project, service, deployment, and log access before approving changes
- **Avoid token setup** - Do not paste Railway tokens or API keys into CloudThinker for this connection
