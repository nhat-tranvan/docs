Connect your Kong Konnect account to enable CloudThinker agents to inventory control planes, audit routes and plugin coverage, and investigate API traffic and analytics across your gateway.

Kong Konnect authenticates with a **Personal Access Token (PAT)**. The token only reaches the resources your Konnect user can view.

---

## Supported Regions

Kong's MCP server is deployed per region. Use the server URL for the region where your Konnect resources live — it defaults to **US**.

| Region | Server URL |
|--------|-----------|
| United States (US) | `https://us.mcp.konghq.com/` |
| Europe (EU) | `https://eu.mcp.konghq.com/` |
| Australia (AU) | `https://au.mcp.konghq.com/` |
| India (IN) | `https://in.mcp.konghq.com/` |

<Note>
A connection reaches one region only. If you run Konnect in multiple regions, create a separate Kong Konnect connection for each.
</Note>

---

## Prerequisites

- A **Kong Konnect account** with access to the control planes you want to investigate.
- A **Personal Access Token** from a Konnect user with permission to view those resources.
- The **regional MCP URL** Kong provides for your region.

<Info>
A Konnect PAT scopes access to what its user can already view. Use a least-privileged user — CloudThinker's discovery, audit, and traffic workflows only read; mutating actions require explicit intent naming the target.
</Info>

## Setup

<Steps>
  <Step title="Create a Personal Access Token">
    In Kong Konnect, open the [Personal Access Tokens](https://cloud.konghq.com/global/account/tokens) page and select **Generate Token**:
    - **Name**: `cloudthinker`
    - **Expiration**: set a rotation window

    Copy the token immediately — it is shown only once.
  </Step>
  <Step title="Get the Regional Server URL">
    From the [Supported Regions](#supported-regions) table, copy the server URL for the region where your Konnect resources live.
  </Step>
  <Step title="Add Connection in CloudThinker">
    Navigate to **Connections → Kong Konnect** and enter:
    - **MCP URL**: the regional MCP URL from Kong
    - **Access Token**: the Personal Access Token you created

    Click **Connect**. CloudThinker verifies the credentials and shows a **Connected** status.
  </Step>
</Steps>

<Warning>
Copy the Personal Access Token immediately after generation. Konnect shows it only once, and you'll need to create a new token if it's lost.
</Warning>

## Connection Details

| Field | Description | Example |
|-------|-------------|---------|
| **KONG_KONNECT_MCP_URL** | Kong's regional MCP server URL | `https://us.mcp.konghq.com/` |
| **KONG_KONNECT_ACCESS_TOKEN** | Konnect Personal Access Token | — |

<Note>
The MCP URL must be an `https://` Kong URL (`konghq.com`); other hosts are rejected.
</Note>

---

## Required Permissions

The Personal Access Token inherits the access of the Konnect user who created it — it can reach only the control planes and resources that user can view.

<Tip>
Follow least privilege: create the token from a user scoped to the control planes CloudThinker needs to investigate. Keep it to read-only access unless a write workflow specifically requires more.
</Tip>

---

## Agent Capabilities

Once connected, agents have read access to your Kong Konnect gateway.

| Capability | Description |
|------------|-------------|
| **Gateway Inventory** | Discover control planes, services, routes, and plugins |
| **Route Security Audit** | Flag unauthenticated exposure and missing auth or rate-limit plugin coverage |
| **Plugin Coverage** | List plugins by control plane and identify services or routes needing protection |
| **Traffic Summary** | Summarize API traffic, failures, and latency over a time window |

<Warning>
Mutating actions — creating, updating, deleting, enabling, or disabling services, routes, plugins, consumers, or control planes — require explicit intent naming the target control plane and resource. They never run during discovery, audits, or traffic summaries.
</Warning>

### Verify the Connection

```bash
@tony inventory our Kong Konnect control planes, services, routes, and plugins so I can understand exposed APIs
```

### Example Prompts

```bash
@tony audit Kong routes for unauthenticated exposure and missing auth or rate-limit plugin coverage and #alert on risky routes
@tony summarize Kong API traffic and failures for the last hour and #visualize error-prone services
@tony list Kong plugins by control plane and #recommend which services or routes need protection
```

## Troubleshooting

<Accordion title="Connection rejected or fails to verify">
The MCP URL must be an `https://` Kong URL on `konghq.com`. Confirm you copied the regional server URL exactly from the Supported Regions table.
</Accordion>

<Accordion title="401 Unauthorized">
The Personal Access Token is missing, invalid, or expired. Generate a new Konnect PAT and reconnect.
</Accordion>

<Accordion title="No control planes or resources found">
The token's user lacks access to those resources, or the URL points at a different region. Use a PAT from a user with access, and make sure the regional URL matches where your Konnect resources are deployed.
</Accordion>

<Accordion title="No traffic or analytics rows">
An empty analytics result is a valid no-traffic answer, not an error. Try a wider time window, or continue with inventory and route audits.
</Accordion>

## Security Best Practices

- **Least-privilege user** - Create the token from a Konnect user scoped to the control planes CloudThinker needs
- **Read-only investigation** - Keep writes out of scope unless a workflow requires them; mutations stay intent-gated
- **Correct region** - Match the MCP URL to the region where your resources live
- **Token rotation** - Set an expiration and rotate the PAT regularly
- **Revoke when unused** - Remove the token in Konnect if you stop using the connection
